PromptBridge — Final Plan

두 계획서의 핵심 문제: MVP 범위 과잉 + 가장 어려운 기술 문제를 과소평가. 이 계획은 그 반대로 간다.

제품 정의 (변경 없음)

비영어권 개발자가 AI 코딩 에이전트를 자기 언어로 쓸 수 있게 해주는 macOS/Windows 데스크톱 브리지.

번역기가 아니라 코딩 컨텍스트 인식 언어 어댑터. 코드·명령어·경로는 절대 건드리지 않음.

가장 어려운 기술 문제 — 먼저 풀어야 함

⚠️ Spike (Week 0): 텍스트 주입 신뢰성 검증

두 계획서 모두 enigo로 "전체선택→복사→붙여넣기" 하면 된다고 가정했는데, 실제로는 다음 상황에서 깨진다:

  • macOS 접근성 권한 없을 때
  • Claude Code / iTerm2 같은 앱이 포커스를 특이하게 처리할 때
  • Windows에서 관리자 권한 앱 위에서 일반 권한 앱이 입력을 못 넣을 때
  • IME 입력 중 단축키가 충돌할 때

해결책: 개발 시작 전 1주일을 이 검증에만 씀. 결과에 따라 전체 아키텍처가 바뀔 수 있음.

진짜 MVP 범위 (7개 기능만)

두 계획서에서 과감하게 제거해야 할 것과 반드시 남겨야 할 것:

✓ MVP에 포함

✓macOS + Windows 시스템 트레이 상주
✓전역 단축키 1개 (입력 번역)
✓플로팅 입력 오버레이
✓한글 → 영어 번역 (DeepL API)
✓클립보드 복사 (붙여넣기는 선택)
✓코드블록 보호 (핵심 차별점)
✓API 키 설정 화면

✗ MVP에서 제거

✗영어→한글 출력 번역
✗프롬프트 최적화 프로필
✗SQLite 캐시
✗LibreTranslate 옵션
✗히스토리 20개
✗자동 엔터 실행
✗민감정보 감지

출력 번역, 프롬프트 최적화, 캐시는 v1.1로. 먼저 입력 하나를 완벽하게 만들어야 함.

현실적인 개발 일정 (8주)

Week 0
기술 검증

Spike — 텍스트 주입 방식 결정

  • macOS: enigo vs AppleScript vs Accessibility API 직접 호출 중 어느 게 Claude Code/터미널에서 안정적인지 테스트
  • Windows: SendInput vs UI Automation API 테스트
  • 결과물: "붙여넣기 방식 결정문서" — 이 결과가 전체 아키텍처를 결정함
Week 1–2
뼈대

Tauri 프로젝트 + 단축키 + 오버레이

  • Tauri 2 + React + TypeScript 프로젝트 생성
  • 전역 단축키 등록 및 충돌 처리
  • 플로팅 입력 창 (frameless, always-on-top)
  • 시스템 트레이 기본 메뉴
  • macOS 접근성 권한 요청 흐름
  • 검증: Claude Code, iTerm2, VS Code, 메모장 위에서 단축키 작동
Week 3–4
번역 + 주입

DeepL 연동 + 텍스트 주입

  • DeepL API Free 연동 (에러 처리 포함)
  • Week 0 결과 기반 텍스트 주입 구현
  • 원래 클립보드 복원 로직
  • 번역 실패 시 UI 피드백
  • 검증: 10가지 한글 입력 시나리오 테스트
Week 5
코드 보호

보호 텍스트 모듈 — 핵심 차별점

  • 코드블록, 인라인 코드, 파일경로, URL, 명령어 탐지
  • placeholder 치환 → 번역 → 복원 파이프라인
  • 검증: "이 git rebase -i HEAD~3 명령어 설명해줘" 같은 혼합 입력 20케이스
Week 6
설정 + UX

설정 화면 + 온보딩

  • API 키 입력 및 저장 (tauri-plugin-store)
  • 단축키 변경 UI
  • macOS 권한 안내 화면 (스크린샷 포함)
  • 자동 시작 옵션
Week 7–8
배포

패키징 + 베타

  • macOS .dmg 빌드 (notarization은 베타 이후)
  • Windows .msi 빌드
  • GitHub Releases + 기본 README
  • 지인 3–5명 베타 테스트 후 v0.1 공개

첫 번째 실행 명령 (오늘)

Week 0 Spike부터 시작

Claude Code 또는 iTerm2를 열고, enigo 없이 순수 AppleScript + osascript로 현재 포커스된 텍스트필드에 텍스트를 주입할 수 있는지 먼저 확인하라. 이게 되면 enigo 기반으로, 안 되면 macOS Accessibility API 직접 접근 방향으로 아키텍처를 잡는다.
